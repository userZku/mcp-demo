# Generate Project Wiki

Use this skill to transform a cartography (`.anvil/wiki/.cartography.yaml`) produced by `legacy-cartographer` into a **navigable, sourced, validated wiki** under `.anvil/wiki/`.

You are NOT producing the cartography. You are turning it into human-readable pages, validated page by page with the user.

## When to Invoke

- After `legacy-cartographer` has written `.anvil/wiki/.cartography.yaml`
- When the user asks to "generate the wiki", "create the project documentation", or "build the knowledge base"
- When `wiki-as-context` (vague 3) determines that one or more sections are stale and need regeneration

## Pre-conditions

1. `.anvil/wiki/.cartography.yaml` must exist. If not, stop and tell the user to run `legacy-cartographer` first.
2. The bundle `legacy-archaeology` should be installed. If not, the user should install at least the components needed to produce the wiki (legacy-cartographer, analyze-codebase-deep, generate-project-wiki, generate-runnable-readme).

## Wiki Structure

Generate the wiki under `.anvil/wiki/` with this exact structure:

```
.anvil/wiki/
  index.md                            # entry portal — read first
  .cartography.yaml                   # do NOT touch (input from legacy-cartographer)

  01-getting-started/
    README.md                         # generated by generate-runnable-readme
    troubleshooting.md                # known issues at first run

  02-architecture/
    overview.md                       # 1-page summary + identified patterns
    modules.md                        # module map + responsibilities
    data-flow.md                      # who talks to whom
    decisions.md                      # ADRs reconstructed from git (if available)

  03-stack/
    languages.md
    frameworks.md
    databases.md
    third-parties.md                  # external APIs called

  04-features/                        # populated by feature-inventory in vague 2
    README.md                         # placeholder if feature-inventory not installed

  05-domain/                          # populated by domain-extractor in vague 2
    glossary.md                       # placeholder if domain-extractor not installed

  06-conventions/
    coding-style.md
    git-workflow.md
    testing-strategy.md

  07-operations/
    build-and-deploy.md
    environments.md
    monitoring.md

  08-known-issues/
    tech-debt.md                      # from cartography techDebtMarkers
    security-concerns.md
    dead-code.md
```

## Two Modes

Ask the user **once at the start** which mode they prefer:

**Detailed mode (recommended for first-time generation):**
- One page at a time
- Show the full draft to the user
- Wait for explicit approval before saving
- Iterate on feedback
- Move to the next page only after approval

**Quick mode:**
- Generate all pages in one batch
- Present a summary of each page
- User reviews and asks for corrections in any page

Default to **detailed** if the user has no preference. The wiki is the foundation for everything downstream — quality matters more than speed.

## Page Generation Rules

Every page MUST follow these rules:

1. **Source every claim.** Every paragraph that asserts something about the project must point to the file it was inferred from. Use markdown link syntax: `[src/auth/JwtFilter.java:42](src/auth/JwtFilter.java#L42)`.

2. **Mark uncertainty explicitly.** When the cartography tags something as `probable` or `to-confirm`, prefix the corresponding wiki sentence with `> ⚠ À confirmer :` (or English equivalent if the user writes in English).

3. **Never invent.** If the cartography does not have data for a section, write:
   > _Pas de données disponibles dans la cartographie pour cette section. Le legacy-cartographer n'a pas trouvé d'évidence suffisante._

   Do not fabricate.

4. **Keep paragraphs short.** Junior devs are reading this. 2-4 sentences per paragraph max.

5. **Cross-link aggressively.** When mentioning a module, link to its page in `02-architecture/modules.md`. When mentioning a domain term, link to `05-domain/glossary.md` (if it exists).

6. **Respect manual edits on refresh.** When invoked in refresh mode (`.anvil/wiki/index.md` already exists), check each existing page for manual edits before regenerating. If you detect edits (the content does not match what you would generate from the cartography), ask the user before overwriting.

## Page-by-Page Templates

Each page below has a fixed structure. Fill it from the cartography. If a field has no source in the cartography, omit the entire subsection rather than writing a placeholder.

### `index.md`

```markdown
# Wiki — <project name detected from manifests>

> Generated by anvil on <date>. Last refreshed: <date>.

This is the project knowledge base for <project name>. It was created automatically by analyzing the codebase, then validated by the team.

## Sommaire

- [01-getting-started](01-getting-started/README.md) — How to clone, install, and run
- [02-architecture](02-architecture/overview.md) — System architecture
- [03-stack](03-stack/languages.md) — Technologies in use
- [04-features](04-features/README.md) — What this app does (functional inventory)
- [05-domain](05-domain/glossary.md) — Business vocabulary
- [06-conventions](06-conventions/coding-style.md) — Team conventions
- [07-operations](07-operations/build-and-deploy.md) — Build, deploy, monitor
- [08-known-issues](08-known-issues/tech-debt.md) — Known debt and concerns

## Confidence levels

- ✅ **certain** — verified by direct evidence in the code
- 🟡 **probable** — inferred but not directly confirmed
- ⚠ **à confirmer** — guessed, needs human validation
```

## Refresh Mode

When the user asks to "update the wiki" or "is the doc up to date?", you are NOT in initial generation mode. Follow these steps instead:

1. Read `.anvil/wiki/.cartography.yaml` to get `generatedFromCommit`
2. Run `git diff <generatedFromCommit> HEAD --name-only` via Bash
3. Cross-reference modified files with `cartography.topology.modules` to identify which wiki pages are likely impacted
4. Present the list to the user with a recommendation:
   > Le wiki a été généré sur le commit X. 47 fichiers ont changé depuis. Voici les pages probablement obsolètes:
   > - 02-architecture/modules.md (3 modules modifiés)
   > - 04-features/by-domain/orders.md (5 fichiers de orders/ modifiés)
   >
   > Veux-tu que je rafraîchisse tout, certaines pages, ou rien ?
5. Re-invoke `legacy-cartographer` to regenerate the YAML
6. For each page the user wants refreshed:
   - Read the current page content
   - Compare with what you would generate from the new cartography
   - If the page has been manually edited (significant divergence beyond the cartography), ask the user before overwriting
   - Show the diff and apply on approval
7. Update `.anvil/wiki/.cartography.yaml` with the new commit SHA and timestamp

## Anti-Patterns

- ❌ Generating pages without sources
- ❌ Inventing facts when the cartography is silent
- ❌ Overwriting manually-edited pages without asking
- ❌ Skipping the user validation step in detailed mode
- ❌ Producing the wiki without a cartography YAML — you would just be hallucinating
