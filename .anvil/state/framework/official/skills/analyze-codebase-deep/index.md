# Deep Codebase Analysis — 4-Layer Method

Use this method when you need to understand a legacy codebase you have never seen before. The goal is **strategic coverage at minimal cost**, not exhaustive reading.

A 200k-LOC codebase cannot be read line by line. The 4 layers are ordered from cheapest to most expensive — you only escalate to the next layer when the previous one is complete.

## Core Principle

> Layers 1-3 are deterministic (Glob, Grep, simple file reads). Layer 4 is the only one that uses expensive AI reading, and it only operates on the **strategic sample** identified by layers 1-3.

If you find yourself reading random files in full, you have skipped a layer. Stop and go back.

## Layer 1 — Inventory (deterministic, fast)

Goal: know what is in the project, by category and by volume.

**Steps:**
1. List all source files via `Glob` patterns:
   - Java: `**/*.java`
   - JavaScript/TypeScript: `**/*.{js,jsx,ts,tsx}`
   - Python: `**/*.py`
   - Go: `**/*.go`
   - C#: `**/*.cs`
   - PHP: `**/*.php`
   - Rust: `**/*.rs`
2. Count files per language and per top-level directory
3. Identify and **exclude** from analysis:
   - `node_modules/`, `vendor/`, `target/`, `build/`, `dist/`, `.venv/`, `__pycache__/`
   - Generated files (look for `// generated`, `# generated`, `<auto-generated>`, or `.g.dart`/`.pb.go` extensions)
   - Lockfiles (`package-lock.json`, `yarn.lock`, `poetry.lock`, `Cargo.lock`)
3. Locate build manifests:
   - Maven: `pom.xml`
   - Gradle: `build.gradle`, `build.gradle.kts`
   - npm: `package.json`
   - Python: `requirements.txt`, `pyproject.toml`, `setup.py`, `Pipfile`
   - Cargo: `Cargo.toml`
   - Go: `go.mod`
   - .NET: `*.csproj`, `*.sln`
4. Compute total LOC (one line per file via `wc -l` is fine, no need to be exact)

**Output of layer 1:** a list of (file path, language, LOC) tuples and a list of build manifests.

## Layer 2 — Topology (deterministic, fast)

Goal: understand the dependency graph between modules without reading any code.

**Steps:**
1. Build the import graph using `Grep` on import statements:
   - Java: `^import\s+`
   - JS/TS: `^import\s+|require\(`
   - Python: `^import\s+|^from\s+`
   - Go: `^import\s+`
   - C#: `^using\s+`
2. For each file, count:
   - **In-degree** (how many other files import it) — high in-degree = central module
   - **Out-degree** (how many other files it imports) — high out-degree with low in-degree = leaf module
3. Detect cycles in the import graph (signal of architectural debt)
4. Find **entry points** — files that have zero in-degree but are executable:
   - Look for `public static void main`, `if __name__ == "__main__"`, `func main()`
   - Look for HTTP route declarations: `@RestController`, `@app.route`, `app.get(`, `router.`
   - Look for cron/job declarations: `@Scheduled`, `cron.schedule(`
   - Look for CLI command declarations: `Command(`, `argparse`, `commander`, `clap`
   - Look for event listeners: `@EventListener`, `subscribe(`, `on(`

**Output of layer 2:** a topology graph with central modules, leaves, cycles, and entry points.

## Layer 3 — Patterns (light semantic, moderate cost)

Goal: identify the architectural style and conventions, **without** reading entire files.

**Steps:**
1. Detect architectural layers via signature `Grep`:
   - Controllers: `@Controller`, `@RestController`, `extends Controller`, `Router`, `@Get/@Post/@Put`
   - Services: `@Service`, `@Component`, `class.*Service`
   - Repositories: `@Repository`, `extends Repository`, `class.*Repository`, `class.*DAO`
   - DTOs/Models: `class.*DTO`, `class.*Model`, `@Entity`
2. Identify the architectural style by counting layer occurrences:
   - Many controllers + services + repositories → MVC / layered architecture
   - Domain/application/infrastructure folders → hexagonal / clean architecture
   - Multiple top-level dirs each with their own controllers + services → microservices or modular monolith
   - Heavy event/message broker usage → event-driven
3. Locate data boundaries:
   - SQL files: `**/*.sql`
   - Migrations: `**/migrations/*`, `**/db/migrate/*`
   - ORM model files (cross-reference with layer 1)
4. List external integrations via `Grep`:
   - Hardcoded URLs: `https?://[^\s"']+`
   - HTTP client usage: `HttpClient`, `RestTemplate`, `axios`, `fetch(`, `requests.`
   - Webhook receivers: `@PostMapping.*webhook`, `app.post.*webhook`
5. Classify tech debt markers:
   - `Grep` for `TODO|FIXME|HACK|XXX|DEPRECATED` and group by file/directory

**Output of layer 3:** architectural style with confidence, layer mapping, external integrations, debt markers.

## Layer 4 — Semantic (AI-powered, expensive)

Goal: understand what the central modules **actually do**. This is the only layer that uses expensive `Read` operations on full files.

**Hard budget: read at most 30 files in this layer.** If you feel the urge to read more, you skipped a layer.

**Steps:**
1. From layer 2, take the top 10-20 most central modules (highest in-degree)
2. From layer 2, take all entry points
3. Read each selected file in full with `Read`
4. For each file, produce a 1-2 sentence summary of its role
5. Extract domain vocabulary from class/method/variable names:
   - Class names like `OrderProcessor`, `CustomerRepository`, `InvoiceLine` → domain concepts: Order, Customer, Invoice
   - Method names like `validatePayment`, `assignContract` → domain operations
6. Pass this vocabulary to `domain-extractor` if it is invoked downstream

**Output of layer 4:** 10-30 file summaries + a flat list of domain terms.

## Stopping Criteria

Stop the analysis when you can answer all of these in 1-2 sentences each, with sources:
- What language(s) and framework(s) does this project use?
- What is the architectural style?
- Where are the entry points?
- Where is the data layer?
- What external services does it talk to?
- What are the main domain concepts (top 10)?

If you cannot answer one of these, you missed a layer or a step. Go back, do not push forward.

## Anti-Patterns

- ❌ Reading files in layer 1 or 2 — those layers are deterministic only
- ❌ Reading more than 30 files in layer 4
- ❌ Pretending to know things you have not verified
- ❌ Skipping the topology layer because "it's obvious from the folder names" — folder names lie, imports do not
- ❌ Trying to understand business logic from code in layer 4 — you only need to understand **role**, not business rules
