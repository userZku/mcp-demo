# Generate Runnable README

Use this skill to produce a "clone, install, run" README that answers the question:

> *I am a new developer on this team. How do I get this app running on my machine?*

## Output Location

**Always** write to `.anvil/wiki/01-getting-started/README.md`.

**Never** write to the project root (`./README.md`). Anvil is a sealed workspace under `.anvil/`. The user is free to copy the file to the root if they want, but this skill never decides for them.

## When to Invoke

- After `generate-project-wiki` has produced the wiki structure
- When the user asks for a "runnable README", "getting started guide", or "how do I run this"

## Pre-conditions

- The cartography (`.anvil/wiki/.cartography.yaml`) should exist — you need to know the build manifests
- The wiki structure should exist — at minimum `.anvil/wiki/01-getting-started/` must be a valid path

## Process

### Step 1 — Detect runnable commands

Read the build manifests identified in the cartography. For each, extract the runnable commands:

- `package.json` → look for `scripts.dev`, `scripts.start`, `scripts.build`, `scripts.test`
- `Makefile` → look for top-level targets (`dev`, `run`, `build`, `test`)
- `pom.xml` → standard Maven goals (`mvn spring-boot:run`, `mvn test`, `mvn package`)
- `build.gradle` → standard Gradle tasks (`./gradlew bootRun`, `./gradlew test`)
- `pyproject.toml` / `setup.py` → look for entry points or scripts section
- `Cargo.toml` → `cargo run`, `cargo test`, `cargo build`
- `go.mod` → `go run .`, `go test ./...`, `go build`
- `*.csproj` → `dotnet run`, `dotnet test`, `dotnet build`

### Step 2 — Detect prerequisites

From the cartography and the manifests, identify:

- **Language runtime version** — search for `engines` in package.json, `<java.version>` in pom.xml, `python_requires` in setup.py, `rust-version` in Cargo.toml, `go` directive in go.mod
- **Required services** — read `docker-compose.yml` if present, list every service block as a prerequisite (databases, queues, caches)
- **Environment variables** — search for `.env.example`, `.env.sample`, or hardcoded `process.env.*` / `os.environ[*]` references in entry points

### Step 3 — Mentally test each command with the user

This is the critical step. For each command you would put in the README, ask the user explicitly:

> Pour vérifier que le README sera vraiment runnable, je vais te poser quelques questions sur les commandes que j'ai trouvées :
>
> J'ai trouvé `npm run dev` dans `package.json`. Est-ce bien la commande qui lance l'app en local sur ta machine ? Y a-t-il des prérequis non évidents (variables d'env, services Docker, secrets) ?

Iterate until each command in the README has been confirmed by the user.

If the user says "I don't know, I never ran this", that's a red flag — the README will probably not be runnable. Note it explicitly:

> ⚠ Cette commande n'a pas pu être validée — un dev qui connaît le projet doit la confirmer avant que ce README soit fiable.

### Step 4 — Detect common first-run errors

Ask the user:
> As-tu rencontré des erreurs au premier lancement de ce projet ? (port déjà utilisé, dépendance manquante, version de Node, secret oublié, base de données non initialisée…)

Capture each one in the troubleshooting section.

### Step 5 — Write the README

Use this exact structure (translate to the user's language):

```markdown
# Getting Started — <project name>

> Wiki section: 01-getting-started · Generated by anvil

This README answers one question: **how do I get this project running on my machine in the next 30 minutes?**

For everything else (architecture, conventions, features), see the [main wiki index](../index.md).

## Prerequisites

<list with versions, e.g. "Node.js >= 18", "Java 17", "Docker">

## Setup

```bash
# Clone (assume already done if you are reading this)
git clone <repo url>
cd <project>

# Install dependencies
<command 1>

# Configure environment
cp .env.example .env
# Edit .env and fill in: <list the keys>
```

## Run

```bash
<run command>
```

You should see <expected output> on <port or URL>.

## Test

```bash
<test command>
```

## Build for production

```bash
<build command>
```

## Common first-run errors

### <error 1 title>

**Symptom:** <message>
**Cause:** <explanation>
**Fix:** <command or instruction>

(repeat for each captured error)

## Need more?

- [Architecture overview](../02-architecture/overview.md)
- [Stack details](../03-stack/languages.md)
- [Conventions](../06-conventions/coding-style.md)
```

### Step 6 — Confirm with the user

Show the full README to the user. Ask:

> Voici le README. Si un junior arrivait demain et suivait ces étapes, est-ce qu'il serait capable de lancer l'app ? Y a-t-il quelque chose qui manque ou qui est faux ?

Iterate until the user is satisfied.

## Hard Rules

- **Never write to `./README.md`** at the project root. Always `.anvil/wiki/01-getting-started/README.md`.
- **Never invent commands.** Every command in the README must come from a real manifest or from the user's confirmation.
- **Never skip step 3 (mental test).** A README that has not been validated with the user is a fiction.
- **Mark unverified commands explicitly.** If a command could not be confirmed, surface the warning in the README itself, not just in your conversation.
